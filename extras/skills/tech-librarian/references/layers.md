# Documentation Layers

Three layers carry the project's knowledge. Each has a different audience, style, and file set.

---

## Layer 1: Sphinx Guides

**Audience:** Human developers, verb authors, game designers.
**Style:** Explanatory prose with worked examples. "Why it works" alongside "what to do."
**Location:** `docs/source/` — organised by Diátaxis category, not numbered files.

| File | Topic |
|------|-------|
| `explanation/introduction.md` | Project overview, what MOO is |
| `explanation/architecture.md` | System architecture, process model, web/registration front-end |
| `explanation/parser.md` | Lexer/parser internals, verb search order, preposition synonym groups |
| `explanation/sandbox.md` | RestrictedPython environment concepts |
| `reference/runtime.md` | Celery tasks, request lifecycle |
| `reference/objects.md` | Object model, inheritance, classes |
| `reference/caching.md` | Property and verb caching |
| `reference/permissions.md` | Permission model, ownership, access control |
| `reference/properties.md` | Property types, inheritance, `inherit_owner` |
| `reference/verbs.md` | Verb model, argument specifiers, permissions |
| `reference/sandbox.md` | RestrictedPython allowed builtins and modules (reference) |
| `reference/parser.md` | Parser reference |
| `reference/tasks.md` | Background task model |
| `reference/builtins.md` | Built-in object hierarchy (`$root_class`, `$thing`, `$player`, etc.) |
| `reference/bootstrapping.md` | Bootstrap process, default world |
| `how-to/creating-verbs.md` | Shebang syntax, dspec/ispec, execution environment, output mechanisms, property access |
| `how-to/advanced-verbs.md` | Calling verbs, time-aware continuation, return values, SDK utilities, best practices |

| `tutorials/building-rooms.md` | Builder onboarding — `@dig`, `@describe`, `@create`, `@alias`, `@obvious` |
| `tutorials/first-verb.md` | First-verb tutorial — Python verb code, shebang syntax, output mechanisms |
| `tutorials/testing-verbs.md` | Testing tutorial — `ContextManager`, `parse.interpret`, `pytest.warns`, locks |
| `tutorials/custom-world.md` | Bootstrap tutorial — `get_or_create_object`, `load_verbs`, `moo_init --sync` |
| `tutorials/first-agent.md` | Agent tutorial — greeter walkthrough + Tradesmen/Mailmen tour |

**Format split:**

| Content | Format |
|---------|--------|
| Guide pages (`docs/source/**/*.md`) | MyST Markdown |
| API reference (auto-generated by autodoc) | RST docstrings in `.py` source files |

Python docstrings must use RST format: `:param x:`, `:type x:`, `:return:`, `:rtype:`, `:raises ExcType:`. Napoleon is **not** installed — Google/NumPy-style docstrings will silently render wrong. When syncing content that touches Python modules, check that any new docstrings follow RST syntax, not Google style.

**What Sphinx does NOT cover:**

- AI agent workflow instructions
- Step-by-step audit passes or skill procedures
- Internal implementation details not relevant to verb authors

## Layer 2: AI Skill Files

**Audience:** Claude Code (AI agent) when a skill is active.
**Style:** Terse, task-oriented, decision rules. Phases and checklists.
**Location:** `extras/skills/*/SKILL.md` plus `references/` subdirectories.

| Skill | SKILL.md focus | Key references |
|-------|----------------|----------------|
| `verb-author` | Writing, editing, testing verbs | `dispatch.md`, `parser-api.md`, `sdk.md`, `testing.md` |
| `game-designer` | Building multi-room environments via SSH wizard commands | `object-model.md`, `moo-commands.md`, `room-description-principles.md`, `verb-patterns.md`, `build-automation.md` |
| `sandbox-auditor` | Security auditing the RestrictedPython sandbox | `audit-history.md`, `attack-categories.md`, `test-patterns.md`, `known-gaps.md` |
| `tech-librarian` | Syncing the three documentation layers | `layers.md` (this file), `translation.md` |

**Skill quality checklist** — when auditing or authoring a skill:

| Check | Rule |
|-------|------|
| Description | Must state WHAT it does AND WHEN to use it. Agents decide whether to activate based on the description alone — vague descriptions cause misses. |
| SKILL.md length | Stay under 500 lines. Move detail to `references/`. |
| `allowed-tools` | Optional frontmatter field (space-delimited) to pre-approve tools for the skill. Example: `allowed-tools: Bash Read Write` |
| Validation | `skills-ref validate ./skill-name/` checks frontmatter and naming conventions. |

**What skills do NOT cover:**

- Human-readable explanations of "why"
- Narrative introductions for newcomers
- Detailed conceptual background (point to Sphinx instead)

## Layer 3: AGENTS.md Files

**Audience:** Claude Code (AI agent) at the start of every conversation.
**Style:** Reference conventions, project rules, quick lookup tables.
**Location:** `AGENTS.md`, `moo/AGENTS.md`, `moo/bootstrap/AGENTS.md`

| File | Scope |
|------|-------|
| `AGENTS.md` | Project-wide: architecture, stack, coding conventions, key entrypoints, testing workflow, security rules |
| `moo/AGENTS.md` | Core module conventions: model patterns, settings, test fixtures |
| `moo/bootstrap/AGENTS.md` | Bootstrap module: default verb patterns, verb naming, test locations |

**AGENTS.md completeness standard** — six core areas from community analysis of 2,500+ repos:

1. **Executable commands** — build, test, deploy commands with full flags (e.g. `uv run pytest -x`, not just "run pytest")
2. **Testing approach** — where tests live, fixtures used, how to run a single test
3. **Project structure** — directory organization, key entry points
4. **Code style** — concrete examples, not just descriptions; include version specifics
5. **Git workflow** — branching, commit message format, PR expectations
6. **Clear boundaries** — what the agent must never touch (secrets, vendor dirs, production configs)

**Limits and quality rules:**

- Maximum 32 KiB (default limit enforced by most AI coding tools)
- Specificity beats vagueness: "uv run pytest -x moo/core/tests/" beats "run the tests"
- Treat as living documentation: update when commands or conventions change
- Avoid duplicating content that is already in skills — link instead

**What AGENTS.md does NOT cover:**

- Step-by-step procedures (those belong in skills)
- Beginner explanations (those belong in Sphinx)
- Content that is already derivable from reading the code

## How to Read Each Layer's Current State

| Goal | How |
|------|-----|
| What does Sphinx say about output? | `Read docs/source/how-to/creating-verbs.md`, search for "print" / "tell" |
| What does Sphinx say about architecture? | `Read docs/source/explanation/architecture.md` |
| What does verb-author skill say about the parser API? | `Read extras/skills/verb-author/references/parser-api.md` |
| What does AGENTS.md say about test patterns? | `Read moo/AGENTS.md` |
| What got added to the SDK recently? | `git log --oneline -- moo/sdk/` |
| What verbs were added recently? | `git log --oneline -- moo/bootstrap/default/verbs/` |
| What autonomous agents exist and what do they do? | `Read extras/agents/README.md`, then each agent's `README.md` |
| How does a specific Tradesman work? | `Read extras/agents/<name>/README.md` and `extras/agents/<name>/SOUL.md` |

## Layer 5: Agent README Files

**Audience:** Human developers setting up or maintaining autonomous agents.
**Style:** Practical setup guide. Purpose, prerequisites, config options, SOUL.md section reference.
**Location:** `extras/agents/README.md` (index) and `extras/agents/*/README.md` (per-agent)

| File | Scope |
|------|-------|
| `extras/agents/README.md` | Index of all agents, agent groups, how directories are structured |
| `extras/agents/foreman/README.md` | Foreman: token orchestrator, $player, stall detection, chain loop |
| `extras/agents/mason/README.md` | Mason: room architect, $player, digs/describes/wires exits |
| `extras/agents/tinker/README.md` | Tinker: object-maker, $programmer, $thing objects and verbs |
| `extras/agents/joiner/README.md` | Joiner: furniture-maker, $player, $furniture and $container |
| `extras/agents/harbinger/README.md` | Harbinger: NPC-summoner, $programmer, ~10% room population |
| `extras/agents/stocker/README.md` | Stocker: consumable/dispenser creator, $programmer |
| `extras/agents/cliff/README.md` | Cliff: pompous postal worker, mail-only entertainment agent |
| `extras/agents/newman/README.md` | Newman: wronged visionary, mail-only entertainment agent |

**What agent READMEs cover:**

- What the agent does in-world and why it exists
- Prerequisites (server access, credentials, LLM provider config)
- How to init a config directory and run the agent
- `SOUL.md` section breakdown: which sections are present and what they control
- Agent-specific config settings in `settings.toml`
- Notable `## Context` links loaded into the agent's system prompt

**What agent READMEs do NOT cover:**

- Game-designer command syntax (covered in `extras/skills/game-designer/references/`)
- Sandbox audit history (covered in `extras/skills/sandbox-auditor/references/`)

Note: per project policy, public Sphinx documentation does not reference
moo-agent or related autonomous-agent tooling. Anything agent-specific
lives in `extras/agents/` and `extras/skills/`, not under
`docs/source/`.
