# Known Quirks

These are pre-existing limitations that are **not bugs** — either canonical
Zork behavior (the game was always like this) or by-design constraints of
the current setup. If you trip one of these, just note it in your
end-of-session summary as "hit known quirk: \<name\>" and move on.

If something in this file feels actionable, move it to [BUGS.md](../BUGS.md)
(game-side fixable) or [TODO.md](../TODO.md) (needs a moo-core change).

## Canonical Zork I behavior

- **`open <obj>` when already-open prints sarcasm ("Look around.", "Have your eyes checked.", …).** Canonical `OPEN-CLOSE` in `actions.zil` calls `<TELL <PICK-ONE ,DUMMY>>` when the object is already open; DUMMY is a table of sarcastic responses by design. Verified against `/Users/philchristensen/Workspace/zork1/actions.zil` `<ROUTINE OPEN-CLOSE>`. The 2026-05-17 shakedown logged this as a bug but it's exactly canonical.
- **The thief steals from the player in dark rooms.** The thief NPC's combat handler calls `_.zork_thing.rob(player, thief)` when player and thief share a dark room (Coal Mine, Bat Room, etc.) with no troll present.  Any inventory item with `tvalue > 0` (treasures, including the broken egg with tvalue=2) can move from the player to the thief without an obvious message — only an "A 'lean and hungry' gentleman just wandered through, carrying a large bag" / "Finding nothing of value, he left disgruntled" / similar narration.  If you notice an inventory item missing mid-session, the thief is the most likely culprit.  Canonical Zork — not a bug.
- **`examine label` returns the full FROBOZZ text, same as `read label`.** Canonical V-EXAMINE in Zork I checks READBIT before falling through to V-READ: if the object has a `text` property, examine prints it. The boat label, leaflet, matchbook, etc. all share this examine/read identity. The 2026-05-17 shakedown initially logged this as a bug but Zork's source confirms it.
- **Trophy case shows contents through closed walls (TRANSBIT).** Canonical `TROPHY-CASE` has flags `CONTBIT TRYTAKEBIT NDESCBIT TRANSBIT SEARCHBIT`. TRANSBIT lets `examine case` see "Your collection of treasures consists of: …" even when closed; `take map` still fails with "You can't reach something that's inside a closed container" because the open-action gate is independent of the read-through gate. Both behaviors are correct.
- **Ancient map starts inside the trophy case at game-start.** The canonical `MAP` object has `(IN TROPHY-CASE)` in `dungeon.zil`. The map's FDESC "In the trophy case is an ancient parchment which appears to be a map." fires on first open. Not a bootstrap bug.
- **Wrench and screwdriver are takeable but not in Maintenance Room desc.** Canonical Zork I's `WRENCH` and `SCREWDRIVER` use the standard "no FDESC + not TOUCHBIT" path through `PRINT-CONT` — items without a first-description default to silent. The room desc lists the tool chests (which DO have FDESC behavior) but not the wrench / screwdriver. Players know where to look from the room's "maintenance room" name. Old BUGS.md entry claiming canonical had them visible was wrong.
- **Egg starts open at game-start (intentional bootstrap shortcut).** `099_reset_state.py` explicitly calls `egg.set_property("open", True)` after placing the canary inside. This bypasses the canonical V-MUNG → BAD-EGG flow that would normally be required to access the canary — the smoke needs the canary takeable via simple `take canary` and the BAD-EGG path replaces it with `broken_canary`, breaking the bauble puzzle. Visible side-effect: `examine egg` shows the canary contents on first visit. Intentional; see the comment block at `099_reset_state.py:135-141`.
- **Chimney up at Studio: the "with what you're carrying" wording is canonical.** `verbs/zork_thing/object_handlers/up_chimney_function.py` mirrors canonical Zork `UP-CHIMNEY-FUNCTION`: empty-handed climb returns "Going up empty-handed is a bad idea." (forces the player to take the painting); carrying ≤2 items including the lantern lets you climb to Kitchen; anything else returns "You can't get up there with what you're carrying." The wording sounds like a capacity check but really means "you must carry the lantern + at most one other item". The 2026-05-11 shakedown initially logged this as misleading but it's the exact canonical Zork I message — not a bug.
- **Mirror Room is two rooms with the same DESC.** Both halves of the rub-mirror teleport print "You are in a circular room … on a strange-looking mirror …" because the bootstrap appends an atom suffix when display names collide — visible as `Mirror Room (MIRROR-1)` / `Mirror Room (MIRROR-2)` in the DB only; the `desc()` verb strips the suffix from player-visible labels. Players distinguish via exits: south-side leads to Narrow Passage / Tiny Cave / Winding Passage; north-side leads to Cold Passage / Small Cave / Twisting Passage.
- **`break mirror` is irreversible after a teleport.** Canonical Zork: rub teleports, break ends the teleport. The slide from Slide Room → Cellar is the one-way escape if you've broken the mirror on the wrong side.
- **`exits` is not a canonical verb.** Canonical Zork I has no `exits` command — it returns "I don't know how to do that." Players use `look` (which describes available exits as part of the room text) or remember the map.
- **`down` / `jump` at Aragain Falls are safe no-ops.** `down` → "It's a long way…"; `jump` → "Wheeeeeeeeee!!!!!" Both responses match canonical Zork: the falls are scenic, not lethal. Player intent to dive in the river is intentionally redirected to the river-launch path (board boat → launch) for the canonical treasure pickup. The 2026-05-17 shakedown flagged these as "should be lethal" but the canonical responses are exactly what ZIL emits.
- **Cyclops puzzle pre-solved at bootstrap (smoke shortcut).** `099_reset_state.py` pre-seeds `zstate_cyclops_flag = True` and `zstate_magic_flag = True` so Living Room west → Strange Passage → Cyclops Room → up → Treasure Room is traversable without driving the maze → ulysses cyclops-feeding sequence. Without the shortcut every smoke run would need to navigate the canonical sequence which currently lacks SDK support for ULYSSES detection. Side effect: chalice / diamond / bracelet / bar / scarab are reachable on first run instead of being post-puzzle rewards. Intentional bootstrap convenience until the maze path lands.
- **"Too late for that." on re-open (canonical DUMMY-RSP).** Canonical `OPEN-CLOSE` (`actions.zil`) picks a random sarcasm from the `DUMMY` table when the player re-opens an already-open object: `Look around.`, `Too late for that.`, `Have your eyes checked.` Same root as the "open already-open" quirk above — wording lifted verbatim from ZIL. The 2026-05-17 shakedown logged this as "non-canonical confusing wording" but the source confirms it's exactly Infocom.
- **`open trap` (bare adjective, no noun) doesn't resolve TRAP-DOOR.** Canonical Zork's ZIL parser requires a noun synonym for object resolution — `TRAP` is an `ADJECTIVE` on TRAP-DOOR, not a `SYNONYM`. `open trap door` / `open trapdoor` / `open door` resolve (`door`, `trapdoor`, `trap-door`, `cover` are synonyms); `open trap` fails the parser exactly as canonical Zork does. Players must include a noun. The 2026-05-17 shakedown flagged this as a missing adjective but the adjective is present — the canonical parser just doesn't accept bare adjectives.
- **`x` is not an examine alias.** Canonical Zork I `<SYNONYM>` declarations don't include `x` for examine — that came in later Infocom games. Use `examine`, `describe`, or `what`.
- **Inventory fumble at 8+ items.** Canonical ITAKE check: when player carries more than 7 items (`FUMBLE-NUMBER=7`), each subsequent `take` triggers a `cnt * 8`-percent fumble roll that prints "You're holding too many things already!" and fails. At 8 items, ~64% chance per attempt; at 9, ~72%. Zork wants the player to drop or repack into the brown sack — it's deliberate game design.
- **`climb walls` fails at End of Rainbow.** The room description mentions cliffs, but canonical Zork has no UP exit there. The CLIMB dispatcher routes `climb <thing>` → `walk("up")`, which fails because no UP exit exists. The cliffs are decorative scenery.
- **Magic boat punctures on anything sharp in inventory.** Boarding while holding the sword (or sceptre, or other sharp items) instantly punctures the boat — "Oops! Something sharp seems to have slipped..." This is canonical and intentional: the puzzle requires you to drop the sword in the Troll Room (after killing the troll) before the boat sequence. Once punctured the boat can't be re-inflated ("No chance. Some moron punctured it.") and the puzzle is unrecoverable for that run.
- **Smoke reset doesn't clean up `punctured boat` Object.** Dam Base shows BOTH `magic boat` and `punctured boat` after a `--reset` if a prior run punctured one. Cosmetic only — `board magic boat` still works (the punctured one is a separate Object).
- **Boat-river drift cadence is tick-precise.** Each player command (including `look`) consumes 1 queue tick; `wait` consumes 4. River segment delays (R1=4, R2=4, R3=3, R4=2, R5=1) define drift timing. The smoke harness's documented cadence (`wait wait look go east`) is OFF by one tick and leaves you stranded at R3 with a failing `go east`. The **working cadence from R1**: `launch` + `look` × 11 + `go east` (lands at SANDY-BEACH with buoy / shovel accessible; smoke baseline 305-317 has never actually landed these treasures). The smoke's "go east at R4 lands at SANDY-BEACH" expectation matches the runtime; its earlier expectation that `look` at R3 drifts to R4 is wrong because R3's speed is 3 and `look` only ticks once. See [BUGS.md](../BUGS.md) entry on the smoke-cadence mismatch.
- **The brown dam button disables the bolt; you must re-push yellow.** Canonical `BUTTON-F` (`actions.zil`): yellow does `<SETG GATE-FLAG T>`, brown does `<SETG GATE-FLAG <>>`. `BOLT-F` reads `GATE-FLAG` and never clears it, so once yellow is pressed `turn bolt with wrench` works on every subsequent Dam visit — until brown is pressed, which is the intended way to deactivate the panel. Verified 2026-05-19: push yellow → turn bolt → leave to the Maintenance Room and back → turn bolt again all succeed. The only writers of `GATE-FLAG` are the four button verbs and the reset (grep-confirmed); nothing else clears it. A 2026-05-19 shakedown logged "bolt stopped turning after returning to the Dam" — that session had pressed the brown button, which is exactly canonical.
- **`push red` in the Maintenance Room has no visible effect on `look`.** Canonical `BUTTON-F` red branch only toggles the room's own `ONBIT`; it never updates the `LIT` global. In real Zork I the player is carrying a lit lantern, so the room stays visible regardless. Our bootstrap also seeds `zstate_always_lit = True` (documented shortcut, see `_reset_state_body.py`), so `is_lit()` short-circuits to true and the room is always described in full. Making the red button darken the room would require dismantling the `ALWAYS-LIT` shortcut (a whole light-source-wiring task), not a targeted fix. The red button is cosmetic by design until then.
- **The boat waterfall death message (`Unfortunately, the magic boat doesn't provide protection...`) fires only at River 5.** It is `i_river.py`'s else branch, reached when `lkp(here, RIVER-NEXT)` is false — true only at RIVER-5, the last `RIVER-NEXT` entry. `board boat; launch; look` at Dam Base cannot show it: after `launch` the i-river daemon is queued at delay 4 (RIVER-1 speed) and a single `look` is one tick, so it never fires. Verified 2026-05-19 in both `zork1_spot` and a live SSH session — the message appears correctly on the fatal drift and the respawn `look` is clean. A 2026-05-19 shakedown logged a "leftover waterfall message at R1"; it was not reproducible and is not a translation gap.

## By-design constraints of this bootstrap / harness

- **"There is a Wizard here." at West of House on the first `look`.** Two player avatars exist on `zork1.local`: `Wizard` (id 5, user `wizard`) and `Adventurer` (id 2082, user `phil`). The shakedown harness connects as `phil`, so you drive the **Adventurer**; `099_reset_state.py` parks *every* site avatar at West of House, so the idle `Wizard` avatar is a genuine co-located occupant the room lists. It does not follow you (the `look` content lister correctly excludes the looker, not the other avatar). Multi-tenancy artifact, not a look-verb bug. Verified 2026-05-20 via DB.
- **`zstate_score` and per-room/per-treasure `value` fields reset on `--sync`.** This is intentional — the smoke needs a known starting state. If you deposit a treasure mid-session and someone runs `moo_init --sync` outside this skill, the score resets.
- **`@quit` is the safe SSH disconnect.** `disconnect()` without `@quit` first leaves a server-side session lingering until the SSH timeout. The harness's `stop` subcommand does `@quit` then `disconnect`.

## Substrate substitutions (player-visible behavior unchanged)

These V-routines are intentionally not generated and have working game-side
replacements. Listed so the next session knows where they live and why
replacement was necessary.

- `V-WALK`, `V-WALK-AROUND` — replaced by `_.walk` exit-Object traversal in `extras/zil_import/verbs/system/movement.py`. The substrate would walk Z-machine memory exit tables we don't model.
- `V-CLIMB-UP`, `V-CLIMB-DOWN`, `V-CLIMB-ON`, `V-CLIMB-FOO` — replaced by the CLIMB dispatcher at [moo/bootstrap/zork1/verbs/zork_actor/dispatchers/climb.py](../../../../moo/bootstrap/zork1/verbs/zork_actor/dispatchers/climb.py): `climb <direction>` → `walk(direction)`, `climb <thing>` → `walk("up")`. So `climb tree` works at Forest Path (UP exit exists); `climb walls` at End of Rainbow fails for the canonical reason above.
- `V-LISTEN`, `V-SMELL` — bare-form fallbacks at `extras/zil_import/verbs/zork_actor/listen.py` / `smell.py` print canonical "You hear / smell nothing unexpected." OBJECT-form syntax dispatches to the substrate via the regenerated dispatcher (which now uses `--dspec any` so bare commands fall through to the fallback instead of crashing the substrate on `None.desc()`).
- `V-ECHO` — bare-form fallback at `extras/zil_import/verbs/zork1/echo.py` prints "echo echo ..." and (if in Loud Room) flips LOUD-FLAG so the platinum bar becomes takeable. The canonical V-ECHO walks parser-internal tables we don't materialize. Lives under `verbs/zork1/` (not `verbs/zork_actor/`) because the Loud Room and bar lookups are Zork1-specific atom names.
- `V-LEAP` — bare-form `extras/zil_import/verbs/zork_actor/jump.py` prints "Wheeeeeeeeee!!!!!" The canonical V-LEAP walks Z-machine exit tables to look for a fall-through direction.

## Verb-row duplication on dispatchers (cosmetic)

`Zork Actor` carries multiple `dispatchers/<verb>.py` files that share the same name set when their underlying ZIL action verbs map to the same V-routine (e.g. PULL/ROLL/YANK all → V-MOVE; PUNCTURE/POKE/DESTROY all → V-PUNCTURE). The parser's "shallowest match wins" picks the lowest-pk row, so duplicates aren't player-visible — but `obj.verbs.count()` shows the extras. Generator dedup is possible but the savings are small (~10 rows out of 350+) and the risk is real (changing dispatcher emission ordering could swap which row wins).
