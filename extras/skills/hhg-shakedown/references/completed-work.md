- **Root cause**: `<JIGS-UP msg>` is in `SDK_HEADS`, so it emits `_.jigs_up(msg)` → the System

  DREAMING/POV-switch branch is dead code for the same reason (callers use `_.jigs_up`, not

- **Fix** (HHG-scoped; Zork untouched by construction):

  3. **Hand-written `verbs/hhg/thing/score/finish.py`**: keeps the score readout, preserves the

## 2026-06-02 — `RUNNING?` checks the real daemon queue → the improbability-drive WIN now fires 🎉

**The single biggest unblock to date: the canonical game ending now works.** Driving the win path surfaced that `<RUNNING? ,I-TEA>` (the gate on SWITCH-F's win branch) was dead.

- **Root cause**: `RUNNING?` is an HHG ZIL routine that walks the `C-TABLE` clock-interrupt table to see if a daemon is scheduled. DjangoMOO doesn't use `C-TABLE` — daemons live in the per-player `zstate_queue` (turn-mode) / System Object `_realtime_pts` (realtime), and the `C-TABLE` globals are **never populated**. So the auto-translated `is_running` always returned **False**, silently killing every `RUNNING?`-gated branch. In `SWITCH-F` the win arm is `<AND <RUNNING? ,I-TEA> <G? ,TEA-COUNTER 6>>` — always False → the player **always** got the failure ending ("moved 8 billion parsecs away…"), never "Good work, kid." Worse, the default emission rendered the daemon atom `,I-TEA` as a **call** (`is_running(_.thing.i_tea())`), executing the daemon as a side effect on every check.
- **Fix** (all HHG-scoped; zork1 has zero `RUNNING?` uses):
  1. **Translator handler** `_h_running` (`translator/expr_handlers.py`, registered `"RUNNING?": _h_running`): emits `_.thing.is_running('<kebab-name>')` — the daemon NAME string, not a call. Mirrors the `_h_queue`/`_h_disable` kebab-name extraction.
  2. **`RUNNING?` added to `_SKIP_ROUTINES`** (`generator/__init__.py`) so the dead C-TABLE version isn't emitted.
  3. **Hand-written `verbs/hhg/thing/predicates/is_running.py`**: checks `zstate_queue` membership by name (turn-mode) with a `_realtime_pts` fallback (realtime). All 24 call sites now read `is_running('i-tea')`, `is_running('i-captain')`, etc.
- **Verified live — the WIN fires**: parked on the post-smoke Bridge (DRIVE-TO-CONTROLS set), forced the remaining win-state via shell (`DRIVE-TO-PLOTTER=T`, `BROWNIAN-SOURCE=TEA`, queued `i-tea`, `TEA-COUNTER=7` — the plotter is unreachable post-ejection so the gameplay half can't be hand-driven from a parked state), then `turn on switch` → *"As you flip the switch, sparks fly… The missiles have turned into a sperm whale at an improbability factor of 2 to the 39,745th power to 1 against… **Good work, kid, says Zaphod**."* The whole deepest-content cascade (FACTOR, the moveto/queue/cancel chain, the MOP/ANNOUNCEMENT/LOST-PLANET string globals via `zstate_get`'s System-Object fallback) dispatches crash-free; post-win `look` is stable. **HHG smoke PASS; zork1 smoke 309/350 Master, 0 crashes (no regression); importer 223.**
- **Behavior note**: this also changes ~12 other `RUNNING?`-gated scenes (appreciate/applaud↔i-captain, stand↔i-prosser, protest↔i-bulldozer, sauna_enter↔i-ford, …) from "always False + daemon side-effect" to real queue state with no side effect — strictly more canonical, and the full smoke (which exercises the Vogon act) stays green.

## 2026-06-02 — Inter-room M-LOOK delegation no longer double-prints the delegate's banner

Fixed the Aft Corridor `look` bug (was in BUGS.md): `look` printed `Corridor, Aft End` (correct) then a spurious `Corridor, Fore End` line, then the description.

- **Root cause**: the translator prepends `print(this.desc())` (the room-name banner) to **every** M-LOOK clause (so rooms with a custom `look` verb still get a heading, since parser dispatch bypasses substrate V-LOOK). AFT-CORRIDOR-F's M-LOOK is `<FORE-CORRIDOR-F ,M-LOOK>` (one room's M-LOOK delegates to another's — FORE-CORRIDOR-F serves both corridors, branching on `player.here() == this`). When invoked as a delegate, the prepended banner printed the *delegate room's* (`this` = Fore Corridor) title even though the player is in the Aft Corridor.
- **Fix** (`moo/zil_import/translator/__init__.py`, both M-LOOK prepend sites — `translate_m_clause` and `_translate_one_m_clause_body`): guard the banner on `if context.player.here() == this:` so it only prints when the routine's owner IS the player's current room. Inter-room delegates (`here() != this`) skip the banner; normal room looks (`here() == this` — describe_room always invokes `look` on `here`) still print it. **Shared translator change** — `context` is auto-imported (matches the `\bcontext\.` pattern); 23 zork1 room-look files picked up the same guard with no behavior change (a normal look is always `here()==this`).
- **Verified**: live on the parked HoG endgame — Aft Corridor `look` now prints a single `Corridor, Aft End` banner + the correct reversed-fore/aft prose; Fore Corridor unchanged. ## 2026-06-02 — Shadowed object-atom references resolve via the atom-map (`_.zatom`)

Closed the deepest open blocker: `turn on spare drive` on the Bridge dispatched to SWITCH-F but the body's `lookup('switch')` grabbed the Galley **dipswitch** (and Hold **case-switch**), both of which carry `switch` as an alias, because moo-core `lookup` matches name OR alias and returns the lowest PK. A ZIL `,FOO` reference always means *the object the author named FOO*, so the right source is the authoritative atom→PK map the bootstrap already records.

- **New System Object verb `_.zatom(ATOM)`** (`moo/zil_import/verbs/system/zatom.py`). Consults `zatom_pk_map` (set by `030_objects.py`) → `lookup(int(pk))`, falling back to the plain alias lookup when the atom is absent (older bootstrap) or recycled. Game-neutral; lives in the shared `verbs/system/` substrate, Rule-Zero-safe (System Object hook).
- **Translator routes only shadowed object atoms through it** (`moo/zil_import/translator/__init__.py` `_translate_atom`, new `ambiguous_object_atoms` param). The object-atom branch emits `_.zatom('FOO')` instead of `lookup('foo')` *only* when FOO's slug is in the ambiguous set; every other atom keeps the plain (peephole-friendly) `lookup(...)`. So the change is a strict no-op for non-colliding atoms.
- **Generator computes the ambiguous set** (`moo/zil_import/generator/__init__.py` `_compute_ambiguous_object_atoms`). Builds a `slug → claimants` map from the same name/alias strings it emits per object (atom slug, display name, action alias, synonyms + expansions), then marks an **object** atom ambiguous when its slug is claimed by >1 object. Candidates are restricted to objects: `zatom_pk_map` is object-keyed, and rooms (created first → lowest PK) always win the `lookup` tiebreak, so a room atom is never shadowed. Threaded to all `ZilTranslator(...)` call sites. Result: HHG reroutes 8 atoms (BED, FLEET, LIGHT, NAME, SLOT, SWITCH, TEA, WINDOW); zork1 reroutes 9 (WATER, LAMP, EGG, KNIFE, BELL, CANARY, BOOK, GUNK, WALL).
- **Verified**: `zatom_pk_map['SWITCH']` → 'generator switch' (pk 6111) where plain `lookup('switch')` returned 'dipswitch' (pk 6096). Live on the parked HoG Bridge: `turn on spare drive` now dispatches to SWITCH-F and prints the canonical "Nothing happens." (the rest of the win — `DRIVE-TO-PLOTTER` + `BROWNIAN-SOURCE`/tea — is unsolved gameplay, not a bug), where before it hit the dipswitch's bogus "Switched. Some lights on the Nutrimat flash…". HHG smoke **PASS**; importer suite 223 passed; zork1 smoke ≥5× within band (see smoke-workflow log).

## 2026-05-30 — Past the babel fish: Vogon poetry trial + airlock

Reaching the post-babel-fish Vogon act for the first time surfaced two turn-model bugs. Both fixed in `moo/zil_import/`; HHG babel-fish smoke still PASS, Zork unaffected (proven below).

- **`goto` VTYPE gate — extract the player from non-vehicle "vehicles"** (`moo/zil_import/verbs/system/movement.py`). The substrate `goto` moved the *vehicle* whenever `current_vehicle()` was non-null. HHG's poetry-appreciation chair has `VEHBIT` (so the player can be strapped into it) but **no `VTYPE`**, so `<GOTO ,HOLD>` / `<GOTO ,AIRLOCK>` dragged the chair along with the player still inside it — the player never became a *direct* occupant of the Airlock, so `AIRLOCK-F`'s per-turn M-END never ticked and the ejection-into-space → Heart-of-Gold rescue never fired. Fix mirrors Zork's own `GOTO` (`zork-substrate/verbs.zil:2045`), whose `.AV` gate moves the vehicle **only when it has a `VTYPE`** (the inflated boat's `NONLANDBIT`); HHG's GOTO (`verbs.zil:2616`) unconditionally moves `PROTAGONIST`. New rule: `if veh and veh.getp("vtype"): move vehicle else: move player`. **Zork-safe by construction** — Zork's only `VEHBIT` object is the boat, which has a `VTYPE`, so the branch is a strict no-op there.
- **HHG `clocker` is a no-op: one daemon tick per turn** (`moo/zil_import/verbs/hhg/thing/helpers/clocker.py`). `do_command` already calls `_.tick()` once at the top of every command; the HHG `clocker` (called only by V-WAIT) ticked *again*, so a `wait` advanced every turn-counter **+2** while other commands advanced +1. The Vogon act is one long daemon cascade gated on exact-equality counters (`CAPTAIN-COUNTER == 6` / `== 11`, `GUARDS-COUNTER == 6`, `AIRLOCK-COUNTER == 4`); a +2 step makes whether a gate lands on a tick boundary depend on the parity of when its daemon armed, so a wrong-parity start steps straight over the gate, it never fires, its daemon is never `DISABLE`d, and the counter runs away (observed CAPTAIN-COUNTER 176, GUARDS-COUNTER 56). Returning `False` (no tick) makes `wait` exactly one turn → counters step +1 → every gate is hit deterministically. zork1's `clocker` keeps the ticking variant (its smoke hand-counts a multi-tick `wait` cadence and it has no exact-counter daemon gate to skip). Verified live: with this fix the poetry counter reaches its gate cleanly (was a runaway). **A residual daemon-lifecycle bug remains past the airlock** — see [BUGS.md](../BUGS.md) "Vogon act daemon lifecycle".
