## 2026-05-24 — First shakedown (Bedroom → Pub)

Five fixes landed mid-session to unblock the opener. Detailed entries in [../BUGS.md](../BUGS.md) under "First shakedown 2026-05-24".

- **`PROTAGONIST` atom resolves to `context.player`** — added entry to `GLOBAL_MAP` in `moo/zil_import/translator/constants.py`. Without this, `<MOVE ,PRSO ,PROTAGONIST>` in V-WEAR teleported worn items into a phantom `it (PROTAGONIST)` Object that the parser can never see. Now `wear gown` works.
- **`PRSO?` / `PRSI?` translator handlers** — added `_h_prso_p` and `_h_prsi_p` in `moo/zil_import/translator/expr_handlers.py`. ZIL's `<PRSO? FOO BAR>` now emits `prso == foo` / `prso in (foo, bar)` instead of an undefined `prso_p(foo, bar)` call. PRE-TAKE and all "is the dobj this specific atom?" branches now work.
- **`PRINTI` translator handlers (stmt + expr)** — added `_h_printi` / `_h_printi_expr`. ZIL's `<PRINTI "lit">` now emits `print("lit", end='')` instead of an undefined `printi(...)` call. Unblocks `score` and any verb that uses PRINTI directly (instead of via TELL). The `<READ>` that follows in `score`/`quit` prompts is still unimplemented — a future fix should route that through `moo.sdk.output.open_input`.
- **`describe_room.py` import switch** — `moo/zil_import/verbs/zork_thing/output/describe_room.py` now imports `NoSuchPropertyError` from `moo.sdk` instead of `moo.core` (which is RestrictedPython-forbidden). Without this every room transition raised `ImportError: Restricted: moo.core`.
- **`describe_room.py` no-op call removed** — dropped the trailing `_.zork_thing.null_f()` from the dark-room branch. HHG's ZIL doesn't define NULL-F, so the call crashed; the call was always a no-op anyway. Zork still has `null_f.py` generated by its translator pass so this doesn't regress.
- **HHG reset seeds light/queue/score state and `player_start`** — `_hhg_reset_state_body.py` now sets `zstate_lit=True`, `zstate_always_lit=True`, `zstate_queue=[]`, `zstate_drop=[]`, `zstate_moves=0`, `zstate_deaths=0`, `zstate_score=0`, and `System Object.player_start = bedroom`. Mirrors Zork's reset; unblocks every dark-room read, the bulldozer death respawn, and several `zstate_get` calls that used to raise `NoSuchPropertyError`.

## 2026-05-24 — Phase 1+2 bug-fix pass (after first shakedown)

Eight more fixes landed after the initial shakedown. All in `moo/zil_import/` (translator, generator, converter, static verb templates, game config). 168 importer unit tests still pass; Zork regenerates with no behavioral diffs (cosmetic-only).

- **ANSWER/REPLY alias collision** — `generator/__init__.py` now post-processes `_ROUTINE_TO_VERBS` to drop any alias on V-X that is the primary name of a sibling V-routine on the same class. Without this, V-ANSWER's `[reply]` alias collided with V-REPLY's primary name and the parser raised `More than one object defines "reply"`. HHG `answer phone` and `reply` now dispatch to V-REPLY cleanly.
- **V-ANSWER P-LEN/P-CONT mistranslation** — `translator/constants.py` `GLOBAL_MAP` now maps `P-CONT`/`P-LEN` to `len(context.parser.words)` and `P-LEXV` to `context.parser.words`. Previously these fell through to `context.player`, producing `parser.words[(player) - 1]` (TypeError at runtime). The new form resolves to `parser.words[-1]` (the last input word) — what V-ANSWER's W?YES / W?NO match actually needs.
- **NULL-F static stub** — added `moo/zil_import/verbs/zork_thing/object_handlers/null_f.py` returning `False`. HHG's ZIL doesn't define NULL-F but the substrate emits `_.zork_thing.null_f()` calls (PRE-TAKE, describe_object, look_inside, …). Zork's translator-generated copy is skipped at the same path via `handwritten_paths` so there's no double registration.
- **Object name fallback** — `generator._compute_display_names` no longer emits `f"it ({atom})"` when the ZIL `DESC` is `"it"` (the canonical "no proper name" idiom for UNIMPORTANT-THING-F objects). Falls back to the first synonym instead — SINK → `basin`, BEDROOM-FURNISHINGS → `carpet`, etc. Player-visible text no longer leaks the placeholder.
- **PTABLE support in converter** — `converter._extract_table_values` now accepts `PTABLE` alongside `TABLE`/`LTABLE`. HHG's INDENTS (a PTABLE of indent strings) was being silently skipped; nested-container `inventory` crashed on `_.table_get(zstate_get("INDENTS"), level)` returning None. Now INDENTS loads onto the System Object and the cascade resolves through the substrate fallback.
- **Per-game synonym expansion** — `GameConfig.synonym_expansions` lets each game register a `truncated → full word` map. ZIL truncates dictionary entries to 6 chars (`ASPIRI`, `ANALGE`, `WASHBA`); `generator._gen_objects` now adds the full word as an extra alias when the truncated form is in the map. HHG's table includes ASPIRI/ANALGE/WASHBA/TELEPH/RECEIV/TOOTHB/SCREWD/CURTAI/BATTER/BUFFER/DRESSI/PROTAG. `take aspirin` now works.
- **Per-game zork_actor filter** — moved Zork's hand-written `verbs/zork_actor/{diagnose,version}.py` into `verbs/zork1/zork_actor/`. The existing top-level `verbs/<other-game>/` skip filter now catches them, so HHG no longer inherits the `ZORK I: The Great Underground Empire` version banner or the `fight_strength` crash on `diagnose`.
- **READ outside REPEAT no longer terminates the verb** — `expr_handlers._h_read` now emits `pass` when `_repeat_depth == 0`, only emitting `return True` inside a `<REPEAT>` loop (where the prior behavior was needed to break LOUD-ROOM-FCN / FINISH REPL loops). The "press ENTER to continue" pattern in V-SCORE / V-QUIT now lets the post-READ code run; the player sees the prompt and then the score display immediately (no actual pause). A future pass should wire `moo.sdk.output.open_input(player, prompt, callback_verb)` for a proper pause-and-resume, splitting the routine into pre-READ and post-READ halves.

## 2026-05-24 — Second shakedown bug-fix pass

Three fixes landed after the second shakedown surface a usable Bedroom loop. 168 importer unit tests still pass; both Zork and HHG regenerate cleanly.

- **V-ATTACK/V-KILL alias collision** — moved the hand-written `moo/zil_import/verbs/zork_thing/substrate_verbs/attack.py` (which has `kill` in its alias list) to `moo/zil_import/verbs/zork1/zork_thing/substrate_verbs/attack.py`. The per-game copy filter excludes `verbs/zork1/` from HHG output, so HHG no longer registers two verbs claiming `kill` on Thing. `kill barman` / `attack barman` now dispatch through the auto-emitted V-KILL substrate's no-op rebuke ("you should learn to relax a little") via the Actor dispatcher's `_.zork_thing.kill()` call. The hand-written attack.py was Zork-specific anyway (uses `lookup("hands")`, `_.jigs_up`, `_.zork_thing.hero_blow()`) — the move clarifies ownership.
- **FINISH self-recursion broken** — added `_READ_RECURSIVE_ROUTINES` allowlist (currently `{"FINISH"}`) in `moo/zil_import/translator/expr_handlers.py`. When the routine being translated is in this set and a recursive self-call appears in the body, the translator emits `None` instead of `_.zork_thing.finish(True)`. The surrounding `_h_default` stmt handler turns the bare `None` into `pass`, so each `<FINISH T>` branch in the else/RESTART-failed/RESTORE-failed paths now exits cleanly. `sleep` (which previously crashed with `RecursionError: maximum recursion depth exceeded`) now runs the full V-SLEEP narrative → JIGS-UP → V-SCORE → FINISH prompt → return chain. The allowlist scope avoids breaking legitimate recursive routines — e.g., WEIGHT calls itself for nested contents, and the previous over-broad version had emitted `wt + None` inside that expression. Adding new routines to the allowlist is safe when the recursion is at tail-statement position.
- **PRE-PUT two-arg HELD?** — extended `moo/zil_import/verbs/zork_thing/helpers/is_held.py` to accept the optional second arg `(obj, container)`. Zork's HELD? takes one arg (defaulting CONT to WINNER); HHG's HELD? has a two-arg form `<HELD? ,PRSI ,PRSO>` used in PRE-PUT to detect circular containment (putting a container into its own contents). The old single-arg helper ignored the second arg, so `is_held(prsi=gown, prso=fluff)` collapsed to "is gown held by player?" → True → false rejection of every `put X in pocket` command. With the fix, the same call walks gown's location chain looking for fluff and returns False, falling through to the canonical V-PUT logic.
- **V-PUT `score_obj` AttributeError** — guarded the `_.zork_thing.score_obj(prso)` call in `moo/zil_import/verbs/zork_thing/substrate_verbs/put.py` with `_.zork_thing.has_verb("score_obj")`. SCORE-OBJ is a Zork routine; HHG has no per-item value scoring, so the auto-translated bootstrap doesn't emit `score_obj.py` on Thing. The bare attribute access AttributeErrored every successful `put` in HHG; the `has_verb` guard makes the score-bump a no-op when the verb isn't present.

After these fixes the canonical Bedroom loop (`take fluff`, `put fluff in pocket`, `wear gown`) works without errors, and `sleep`/V-SCORE no longer crash. The earlier session's `kill barman` / `attack barman` errors are gone.

## 2026-05-24 — Third bug-fix pass (interleaving + scenery resolution)

Two structural fixes that cleared up many text-ordering and dobj-resolution issues.

- **Print-call interleaving (ARTICLE)** — replaced the auto-translated `verbs/zork_thing/helpers/article.py` with a hand-written version that RETURNS the article string instead of calling `print()`. Added a new `_h_article` statement handler in `moo/zil_import/translator/stmt_handlers.py` that emits `<ARTICLE obj the>` as `print(_.zork_thing.article(obj, the), end='')`. The translator's existing `_merge_adjacent_prints` peephole then folds consecutive TELL + ARTICLE + TELL sequences into a single `print("A" + article(prso, True) + "B" + ..., end='')` — so the article output now lands in the caller's print buffer instead of triggering a sub-verb buffer flush. Visible effect: `put fluff in pocket` → "Done." (was broken), `wear gown` → "You're already wearing the your gown!" (was article-first/parent-last interleaving), `answer phone` → "It is hardly likely that the telephone is interested." (single sentence), V-PUT error messages render in the right order. Still open: `<TELL ...> <JIGS-UP ...>` patterns (bulldozer death message) — JIGS-UP can't easily return a string because it has state-mutation side effects. The pattern needs a translator peephole that prepends the preceding TELL text into the JIGS-UP arg.
- **Scenery atom → object resolution disambiguation** — added a `zatom_pk_map` System Object property emitted at the end of `030_objects.py`: `{atom: obj.pk for atom, obj in _objects.items()}`. Updated `verbs/system/resolve_dobj_late.py` to consult the map FIRST when resolving a scenery atom (e.g., the WINDOW atom in Bedroom.global_scenery). Without the map, `lookup("window")` returned CANOPY (because CANOPY's SYNONYM list includes WINDOW, putting an alias collision on the same name). With the map, the WINDOW atom resolves to PK 3473 (the actual window object in LOCAL-GLOBALS). `examine window` from the bedroom now triggers the WINDOW-F's M-LOOK branch ("the sun is shining, the birds are singing, ... a large yellow bulldozer is advancing on your home"). `examine house` from Back of House now correctly says "There's nothing special about the house" (was "the it" because of duplicate-objects-on-sync + missing atom disambiguation).
- **`buy <item>` is canonical HHG behavior** — investigated and confirmed: when player tries `buy beer` (as Arthur, beer has NDESCBIT meaning Ford has already supplied beer), `verbs/rooms/pub/beer/buy.py` does `_.perform('buy', lookup('peanuts'), None)` — redispatches to peanuts. peanuts has no matching buy branch for Arthur identity → `passthrough()` → V-BUY → "Sorry, the peanuts isn't for sale." This is the HHG game design (Arthur shouldn't be ordering beer in the Pub at game start). Not a bug; moved out of BUGS.md.
- **`LIE DOWN` (bare) is canonical Z-machine refusal** — investigated and confirmed: HHG's syntax has `<SYNTAX LIE DOWN OBJECT>` (requires OBJECT after DOWN) and no bare `<SYNTAX LIE>`. Z-machine parser refuses bare `lie down` because no syntax matches. The compound dispatcher in `verbs/zork_actor/dispatchers/lie.py` handles `lie down <object>` but parser-level dispatch requires the OBJECT first. Same shape for `stand up`. Not a translator/generator bug; HHG canonically requires `lie in mud` / `lie on bed` / `stand on chair`. Moved out of BUGS.md; could land as an explicit known-quirk if it comes up again.
